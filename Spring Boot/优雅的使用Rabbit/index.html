优雅的使用Rabbit

Last updated on June 24, 2023 pm

介绍

以前使用 RestTemplate 发起远程调用的代码存在一些问题，比如可读性差、编程体验不统一、复杂的参数和难以维护的 URL。为了解决这些问题，我们可以使用 Feign，它是一个声明式的 HTTP 客户端，可以优雅地发送 HTTP 请求。

在微服务项目中，使用消息队列进行解耦也会遇到类似的问题，包括可读性差、编程体验不统一、复杂的参数和难以维护的消息处理逻辑等。

问题一：对于十几种不同的通知消息，我们可以采取以下方式确保消息执行正确的逻辑：

  • 可以为每种消息创建一个独立的队列，每个队列对应一种消息类型。
  • 可以将几个消息归类到同一个队列中，然后根据消息类型在接收方进行判断，执行相应的逻辑代码。

问题二：如果 user 需要接收一种新的消息，可以采取以下方式处理：

  • 为 user 添加一个新的队列，专门用于接收这种新的消息。
  • 如果这种新消息可以归类到已有的队列中，可以在同一个队列下根据消息类型进行判断，执行不同的逻辑。

问题三：关于消息携带的信息，有以下几种方式：

  • 对于不同类型的消息，可以创建不同的消息对象（类），接收方根据消息对象类型进行接收和处理。但是随着消息种类的增多，这种方式可能变得难以维护。
  • 可以统一将消息对象设置为 Map，消息发送方将需要的参数放入 Map 中，接收方根据参数键获取相应的值。但是这种方式也可能导致维护困难。

以上是一种简化的解决思路，但确实存在一些问题。为了解决这些问题，我设计了一个工具包（mqtool），它基于 Java 注解、Spring AOP 和 Java 反射来简化消息的处理。使用 mqtool，只需进行简单的配置，就能实现类似 Feign 的功能。发送方通过注解指定 URL，接收方也通过注解指定 URL，它们之间建立了一对一的关系。问题一和问题二得以解决。至于问题三，需要保证发送方的参数和接收方处理方法的参数保持一致，MQ 会帮助我们获取参数并进行处理，简化了使用过程。

除了上述功能外，mqtool 还提供了其他细节化的功能，可以进一步优化和简化代码，使其易于理解和使用。

基本使用

拉取项目 xtwoy/MQ (gitee.com) 打包 mqtool ，加入本地maven仓库

使用MQ的准备工作

  1. 在这之前准备好了rabbit的exchange queue

    在 ProjectQueueConfig UserQueueConfig 中。

  2. 注册发送客户端

    UserClient.java

    1. @MQClient 注解：

      • 用于注册一个消息发送客户端。
      • 使用 serviceName 属性设置接收服务端的名称。

    2. @MQSend 注解（用于类上）：

      • 标记该类为一个消息发送者。
      • url 属性表示该客户端下发送消息的路径，类似于控制器中的 RequestMapping。
      • 所有消息体的 URL 路径都以 url 属性的值为开头。

    3. @MQDirectSend 注解：

      • 表示消息将发送到一个 direct 类型的交换机。
      • 使用 url 属性设置消息体的 URL。
      • UUID 属性将使用 Spring EL 进行解析，类似于 @Cacheable 中的 key 属性。它将作为消息体的唯一标识。

  3. 监听消息，并消息体转交消息处理工厂处理

    Consumer.java

    @MQCommit 修饰的方法，其中一个参数的类型为 MQMsg，表示该方法用于接收消息，并将消息交给处理工厂进行处理。

    在消息到达时，由 @MQCommit 修饰的方法进行处理。其中，MQMsg 参数表示接收到的消息对象。通过该参数，您可以访问消息的内容和元数据。

    处理工厂是一个组件，用于处理不同类型的消息。根据消息的类型或其他标识，处理工厂可以根据需要选择执行相应的处理逻辑。它负责解析消息内容，并根据业务需求执行相应的操作。

    通过将消息交给处理工厂处理，可以实现消息的分发和处理逻辑的解耦。这样，您可以根据具体的业务需求，动态添加、修改或删除消息的处理逻辑，而无需修改接收消息的方法。

    通过以上方式，您可以更灵活地处理不同类型的消息，并根据需要选择执行相应的处理逻辑，从而提高代码的可扩展性和可维护性。

  4. 配置 exchange url route-key之间的映射关系。

    MQTool 是如何将消息发送到正确的队列的呢？

    1. MQTool 需要获取以下信息来确定消息发送的正确队列：

      • 服务名称（serviceName）：需要知道将消息发送到哪个服务上。
      • 交换机类型（type）：需要知道消息应该发送到什么类型的交换机上。
      • 消息的 URL（url）：需要知道消息的具体 URL 是什么。

    2. 基于以上三个信息，MQTool 可以配置路由：

      • 首先，根据 serviceName 找到与该服务相关的路由配置信息。
      • 然后，根据 type 找到与该类型相关的路由配置信息。
      • 最后，根据 url 找到具体的路由配置。可以使用精确的 URL，也可以使用 Ant 风格的匹配。

    3. RouteName 是一个数据结构，包含以下三个字段：

      RouteName 记录了路由的相关信息，包括要发送到的交换机名称、URL 路径和 routingKey。

    通过以上方式，MQTool 可以根据服务名称、交换机类型和消息的 URL 来确定正确的队列，并将消息发送到相应的队列中。这样可以实现消息的准确路由和分发，确保消息被正确地处理和消费。

  5. 注册消息处理者。

    UserWork.java

    1. @MQWork 注解：

      • 用于注册一个消息处理工作器（Worker）。
      • 使用 url 属性设置消息处理工作器要处理的消息的 URL，类似于 @MQSend 中的 url 属性。

    2. @MQReceive 注解：

      • 用于标记方法作为消息的接收者。
      • value 属性表示要接收的消息的 URL，类似于 @MQSend 中的 url 属性。

    3. 不需要关注如何从消息体中获取方法的参数，mqtool 会自动帮助解析，类似于 Feign 远程调用中自动解析参数的机制。

    4. 在 mqtool 中，一个被 @MQWork 修饰的 Bean 对象被视为一个消息处理工人（Worker）。

      • 该工人具备处理消息的能力。
      • 例如，一个被 @MQWork(url = "/user/demo") 修饰的工人可以处理消息体中 URL 为 “/user/demo” 的消息。
      • 类似地，工人可以根据 @MQWork 注解上设置的 URL 来处理对应的消息。

    通过以上方式，可以将消息的处理逻辑与具体的方法解耦，并通过 @MQWork 和 @MQReceive 注解来标记消息的处理者和接收者。这样可以更加清晰地定义工作器和消息接收者的能力，并实现灵活的消息处理和分发。

总结

使用mqtool，还是存在多队列维护麻烦的问题。按照我现在的理解，一个服务不一定只监听一个服务。所以必须手动的进行配置。通过配置之后。只要保证消息发送它该去的地方之后，你不需要关注消息如何得到正确的处理，如何从消息挈带信息中获取我们需要的信息，而是关注消息真正的处理逻辑。

mqtool现在也用在我的毕设里面。效果还不错的样子。王婆卖瓜，还自卖自夸。👏👏👏

mqtool虽然就实现了一些很简单功能。也是我制作的第一个工具jar包。也花了不少的时间。😊😊😊

尽管整体项目设计可能存在一些问题，导致了代码的反复修改，但这也使我更加熟悉了Java注解、Spring AOP和Java反射等技术。这是宝贵的学习经验。
