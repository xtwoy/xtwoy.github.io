优雅的使用Rabbit

Last updated on June 16, 2023 pm

介绍

以前 RestTemplate发起远程调用的代码，存在代码可读性差、编程体验不统一、参数复杂URL难以维护等问题。

通过Feign，可以很方便的进行服务间的调用。Feign是一个声明式的http客户端，其作用就是帮助我们优雅的实现http请求的发送。

同时微服务项目使用消息队列进行解耦，也会存在代码可读性差、编程体验不统一、参数复杂、消息该使用什么代码处理难以维护等问题。

比如，一个微服务项目中，有project，user两个服务。

问题一：两个服务之间有十几种不同的通知消息，如何想让这些消息都正确地去执行这个消息本应该执行的逻辑？

  1. 一种消息，对应一个队列？
  2. 几个消息归为一类，在根据消息类型，判断执行什么逻辑代码？

问题二：现在user又要增加接收一种消息，如何处理？

  1. user消息接受方增加一个队列？
  2. user在同一个队列下面，根据消息在来判断执行不同的逻辑？

问题三：消息，消息，肯定要携带一些信息。这些信息怎么携带？

  1. 不同类型的消息，放入不同的消息对象（class），接收方去接收对应的消息？但是因为消息种类复杂，难以维护。
  2. 将消息对象通统一设置为map？需要什么参数，直接去get(need)。同样存在难以维护。

问题……

也是自己写做项目遇到以上的问题，主要通过Java注解、Spring Aop、Java反射来解决这些问题，后面设计为工具包–MQ，方便使用。

使用MQ，完成一个配置，就可以如同Feign一样。你只需要在发送方通过注解指定URL，接收方通过注解指定URl，他们之间便有一一对应的关系。问题一，二，得以解决。关于问题三，需要保证发送方的参数和接收方的处理方法的参数一致。MQ会帮你完成参数的获取，直接使用就ok。

还有其他细节化的功能……

基本使用

先将jar包添加到maven本地仓库

使用MQ的准备工作

  1. 在这之前准备好了rabbit的exchange queue

    在 ProjectQueueConfig UserQueueConfig 中。

  2. 注册发送客户端

    UserClient.java

    1. @MQClient
      1. 注册一个消息发送客户端。
      2. serviceName 设置接收服务端名。
    2. @MQSend(类上)
      1. url（uri-a），在该客户端下发送的消息体。url路径都以url-a开头。参考控制层 RequestMapping
    3. @MQDirectSend
      1. 消息将发送到direct类型的交换机上。
      2. url属性设置消息体的url
      3. UUID会使用Spring EL 解析。参考@Cacheable中的key。会作为消息体的唯一标识。

  3. 监听消息，并消息体转交消息处理工厂处理

    Consumer.java

    @MQCommit修饰的方法，有个参数的类型为MQMsg。在收到消息，会将消息交给处理工厂处理。

  4. 配置 exchange url route-key之间的映射关系。

    1. mqtool如何将消息进行发送正确的队列？

      1. 需要知道发送到**那个服务（serviceName）**上
      2. 需要知道发送到什么类型的交换机（type）
      3. 需要知道消息的url是什么
      4. 通过以上三个信息，配置路由。
        1. 然后可以通过 serviceName 找到关于 serviceName服务 的 路由配置信息。
        2. 在根据 type 找到关于 type 的路由配置信息。
        3. 在根据 url ，找到具体是哪一个路由配置。可以使用精确的url，也可以使用Ant风格。

    2. RouteName

      1. 有三个字段。

  5. 注册消息处理者。

    UserWork.java

    1. @MQWork
      1. 注册一个消息处理work
      2. url属性，作用于上面的MQSend 类似。
    2. @MQReceive
      1. value 属性，类似上面的MQSend的url属性。
    3. 不需要关注怎么从消息体中去获取，方法的参数。mqtool会自动帮助解析。参考Feign远程调用。
    4. 在mqtool中，一个被MQWork修饰的bean对象，被当作一个打工人。
      1. 他具有处理消息的能力，上面就有处理 消息体中url为**"/user/demo"** 或**"/user/demo1"或"/user/demo2"** 或**"/user/demo3"** 的能力

总结

使用mqtool，还是存在多队列维护麻烦的问题。按照我现在的理解，一个服务不一定只监听一个服务。所以必须手动的进行配置。通过配置之后。只要保证消息发送它该去的地方之后，你不需要关注消息如何得到正确的处理，如何从消息挈带信息中获取我们需要的信息，而是关注消息真正的处理逻辑。

mqtool现在也用在我的毕设里面。效果还不错的样子。王婆卖瓜，还自卖自夸。👏👏👏

mqtool虽然就实现了一些很简单功能。也是我制作的第一个工具jar包。也花了不少的时间。😊😊😊

项目整体的设计，并不是很好。导致好一部分代码，都是改来改去的。也有对偏底层不熟悉的缘故。

还是学习到一些东西，起码对于Java注解、Spring Aop、Java反射有了更进一步的理解。
